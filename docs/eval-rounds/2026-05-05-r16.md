# Round R16 (2026-05-05)

First round shipping audit-driven *deletions* rather than additions.
Two of three planned deletions shipped: with_gla skip for source-less
scenarios, and full revert of the budget hint from advisor +
config_advice. Run partially invalidated by claude-cli rate-limit
mid-cohort; rerun queued.

## Ran

- Cohort: same 14 R12 scenarios
- Modes: `with_gla` (skipped — all 14 are source-less),
  `code_only` (3 of 14 ran before rate-limit)
- Model: `claude-opus-4-7[1m]` via claude-cli
- Output: `/data3/gla-eval-results/2026-05-05-r16/`
- Boundary: `round-r16-start` (`4e7b32e`) → `round-r16-end` (`680c65a`)

## Findings

- **with_gla deletion worked perfectly.** All 14 source-less scenarios
  skipped instantly (`Saved 0 result(s)` in 1 s, $0 cost). The
  comparison they were producing was theater pre-R16; deleting the
  runs saved ~50% of cohort time and judge cost without losing any
  signal.
- **claude-cli rate-limit hit mid-run** at scenario 4. First 3
  scenarios ran legitimately; remaining 11 got "You've hit your
  limit · resets 10pm (Asia/Seoul)" as their diagnosis_text. Pre-R16
  the harness silently scored these as scenario failures with 0
  tokens — 11 fake data points contaminating the cohort.
- **Process gap surfaced**: this is exactly the kind of issue the
  Musk 5-step audit is designed to find. We'd been operating with
  silent partial-data corruption (R15's numbers may have been
  affected too — would need to grep diagnoses for the rate-limit
  signature retroactively). Fix shipped (`680c65a`): cli_agent now
  raises `CliRateLimitError` on the signature, the CLI catches it,
  saves partial results, and exits with code 3.
- **Of the 3 scenarios that did run**: 0/3 solved. Within stochastic
  noise (R15 had godot_glow_extremely_slow solving in code_only;
  R16 same scenario failed). Insufficient sample to validate the
  budget-hint revert hypothesis.

## System-improvement audit (Musk 5-step)

**1. Requirements less dumb.**

- **"Eval cohort runs are atomic — partial results are noise."**
  Owner: original eval design. Still load-bearing? **No** — partial
  results from a 3-of-14 rate-limited run are still useful for
  forensic analysis if persisted with provenance. Pre-R16 the run
  saved 14 result rows including 11 fakes. R16's fix saves only the
  3 real ones with a clear `[rate-limit]` exit. Better design.
- **"with_gla mode is meaningful for every scenario."** Owner: me
  (R10 era). Still load-bearing? **No** for source-less scenarios,
  re-confirmed by R16's clean skip. Done.

**2. Deletions.**

- **with_gla runs for 13/14 R12 cohort scenarios** — deleted (`4e7b32e`).
  Cost saving: ~50% of per-cohort compute + judge calls.
- **Budget hint from advisor + config_advice prompts** — deleted
  (`4e7b32e`). Effect on solve rate pending validation.
- **Silent rate-limit-as-failure recording** — deleted (`680c65a`).
  Replaced by hard error + partial-save + clear exit.

**3. Simplifications.**

(none this round — would have been P2/DiagnosisScorer cleanup;
deferred)

**4. Cycle-time improvements.**

- The with_gla skip cuts cohort runtime ~50% on source-less cohorts.
  R16's partial run hit limit at scenario 4 instead of scenario 8 it
  would have hit pre-deletion. Net: deletions also accelerate cycle
  time as a side effect.

## Side observations

- Rate-limit fix is the kind of robustness improvement audits surface
  but optimization-first thinking misses. We'd been adding scoring
  fixes for months while silently recording rate-limit error pages
  as bad diagnoses.
- The rerun discipline matters: rather than declaring R16's 0/3 a
  regression, we treat it as invalidated and re-run. Saves a wasted
  forensic cycle.
- R16 is the first round to ship deletions as the primary action.
  Next round should produce more — the system is still over-engineered
  in places (3 prompt templates, multiple verifier tiers).
